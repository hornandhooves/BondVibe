# KQA-003 — MatchPersonScreen truena sin el param `profile` (falta guard)

- **Severidad:** P4 (robustez / edge case)
- **Estado:** Nuevo (opcional)
- **Pantalla / Flujo:** matching/MatchPersonScreen
- **Entorno:** detectado por `smoke.subfolders.test.js`
- **Fecha:** 2026-07-17

## Resumen
`MatchPersonScreen` lee `profile.photoUrl` (L73) sin guard sobre el objeto `profile`. Si se navega sin el param `profile`, truena con `Cannot read properties of undefined (reading 'photoUrl')`. Es la única de 63 pantallas de subcarpetas sin guard del param raíz (sus campos individuales sí están guardados; el objeto no).

## Contexto
En el flujo normal siempre se navega con `profile` (desde el grid de match), así que un usuario real no lo topa. Lo levantó el smoke recursivo. Por eso es P4: consistencia/robustez, no un crash en uso normal.

## Causa (código)
`src/screens/matching/MatchPersonScreen.js` L26 y L73:

    const { eventId, eventTitle, profile } = route.params || {};   // profile puede ser undefined
    ...
    {profile.photoUrl ? ( ... )}                                   // sin guard sobre profile → truena

## Fix sugerido (dev, opcional)
Guard temprano, alineado con el resto de la app:

    const { eventId, eventTitle, profile } = route.params || {};
    if (!profile) return null; // o un fallback / navigation.goBack()

## Nota
El smoke ahora le pasa un `profile` de prueba, así que la suite queda verde; este ticket sólo rastrea la mejora de robustez.
